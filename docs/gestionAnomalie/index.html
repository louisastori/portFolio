M2L – Réservation de salles – Ticketing

Dans le cadre de ce projet, nous avons travaillé sur un site web existant pour lequel une série d'améliorations fonctionnelles nous a été demandée. Ces demandes étaient formulées sous la forme d’une douzaine de user stories, c’est-à-dire des descriptions courtes et centrées sur l’utilisateur, exprimant un besoin ou une fonctionnalité à implémenter.

Nous avons réparti ces user stories entre les membres de l’équipe, puis nous les avons traitées une par une. Une fois une user story terminée, nous la cochons pour indiquer son avancement.

En revanche, nous n’avons pas estimé le temps prévu pour chaque user story, ni mesuré le temps réellement passé.

Site Web Dynamique

Dans le cadre de cette AP, nous avions pour objectif de créer un site web dynamique permettant de fournir des informations sur les intervenants, leurs contrats, ainsi que d'autres fonctionnalités spécifiques demandées par le client.

Le projet nous a été présenté sous la forme d'une liste d'exigences fonctionnelles, que nous avons reformulées sous forme de user stories.

Nous avons ensuite transformé ces user stories en tickets de travail, que nous nous sommes répartis. Contrairement à un projet précédent, nous avons ici estimé et suivi le temps par ticket.

Cela nous a permis d’évaluer notre avancement, d’ajuster notre organisation, et de mieux anticiper les difficultés.

Sur cette AP nous avons choisi de faire un commit général à la toute fin du projet. Après réflexion ce choix n’était pas bon, nous aurions dû faire des commits et des merges plus régulièrement.
Fiche JIRA

Durant mon stage, j’ai participé à un projet de développement encadré par une méthode agile. Pour organiser et suivre l’avancement du travail, l’équipe utilisait des fiches JIRA.

Chaque fiche JIRA correspondait à une user story. Ces fiches comportaient une description claire, une estimation du temps, et étaient discutées lors des réunions de grooming.

Cependant, ces réunions n’étaient pas toujours tenues à chaque sprint. Cela a pu entraîner une gestion du temps moins précise.

Dans chaque ticket JIRA, on retrouve une description fonctionnelle, des critères d’acceptation, priorités, et parfois des captures d’écran.

Le Product Owner rédige les tickets, les prépare et les intègre avec toutes les informations nécessaires.

Une bonne rédaction permet d’éviter les malentendus, de gagner en efficacité, et de mieux cadrer le travail du développeur.

J’ai constaté que la qualité des tickets influençait directement la fluidité du développement.

Gestion des Anomalies

Pendant mon stage, j’ai participé à la gestion des anomalies. Chaque matin, les nouvelles anomalies étaient listées dans une section « Nouveau » du Kanban.

Le capitaine Anno analysait et attribuait ensuite les anomalies aux développeurs.

Une fois corrigées, elles passaient en test. Si la testeuse validait la correction, elles étaient marquées comme « Terminée ».

L’équipe utilisait deux Kanban : un pour les fonctionnalités et un autre pour les anomalies. Cette séparation facilitait la gestion parallèle du développement et du support.
